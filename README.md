## рџ”Ґ Firebase Setup Guide

### Step 1 вЂ” Create a Firebase Project

1. Go to [console.firebase.google.com](https://console.firebase.google.com)
2. Click **Add project** and name it `kigali-city-services`
3. Enable **Authentication** в†’ Sign-in method в†’ **Email/Password**
4. Create a **Firestore Database** in **Production mode**, region `europe-west1`

### Step 2 вЂ” Register your Android App

1. In Firebase Console, click the Android icon to add an app
2. Enter your package name вЂ” find it in `android/app/build.gradle` under `applicationId`
3. Download `google-services.json` and place it in `android/app/`

### Step 3 вЂ” Connect Flutter to Firebase

Install the FlutterFire CLI and run configuration:

```bash
dart pub global activate flutterfire_cli
firebase login
flutterfire configure
```

This will automatically generates `lib/firebase_options.dart` with your project credentials and edits the Gradle files. You do not need to manually edit any Gradle file.

### Step 4 вЂ” Google Maps API Key

1. Go to [console.cloud.google.com](https://console.cloud.google.com)
2. Enable **Maps SDK for Android**
3. Create an API key under **APIs & Services в†’ Credentials**
4. Add it to `android/app/src/main/AndroidManifest.xml`:

```xml
<meta-data
    android:name="com.google.android.geo.API_KEY"
    android:value="YOUR_API_KEY_HERE"/>
```

### Step 5 вЂ” Firestore Composite Indexes

Create these two indexes in Firebase Console в†’ Firestore в†’ Indexes:

| Collection | Field 1 | Field 2 | Scope |
|------------|---------|---------|-------|
| listings | createdBy (Asc) | createdAt (Desc) | Collection |
| reviews | listingId (Asc) | createdAt (Desc) | Collection |

### Step 6 вЂ” Run the app

```bash
flutter pub get
flutter run
```

## рџ—„пёЏ Firestore Collections

The database has three top-level collections.

### `users/{userId}`

Created automatically when a user registers. The document ID is the Firebase Auth UID.

```
userId      String   вЂ” Firebase Auth UID
email       String   вЂ” user's email address
name        String   вЂ” display name
createdAt   Timestamp
bookmarks   Array<String>  вЂ” list of bookmarked listing IDs
```

### `listings/{listingId}`

Each document shows one place in the directory. Document ID is auto-generated by Firestore.

```
id              String   вЂ” auto-generated document ID
name            String   вЂ” name of the place
category        String   вЂ” Hospital | Police Station | Library | Restaurant | CafГ© | Park | Tourist Attraction
address         String   вЂ” human-readable address
contactNumber   String   вЂ” phone number (optional)
description     String   вЂ” longer description
latitude        Number   вЂ” GPS latitude (set via map pin picker)
longitude       Number   вЂ” GPS longitude (set via map pin picker)
createdBy       String   вЂ” UID of the creator
createdByName   String   вЂ” display name of the creator
createdAt       Timestamp
rating          Number   вЂ” aggregate average rating (0.0вЂ“5.0)
reviewCount     Number   вЂ” total number of reviews
```

### `reviews/{reviewId}`

Each document represents one user review on a listing.

```
id          String   вЂ” auto-generated document ID
listingId   String   вЂ” references the parent listing
userId      String   вЂ” UID of the reviewer
userName    String   вЂ” display name of the reviewer
comment     String   вЂ” written review text
rating      Number   вЂ” star rating (1.0вЂ“5.0)
createdAt   Timestamp
```

When a review is submitted, the app recalculates the average rating across all reviews for that listing and writes the updated `rating` and `reviewCount` back to the listing document immediately.

## рџ”„ How Data Flows вЂ” From Firestore to the UI

This is the most important architectural concept in the project. Data always travels through the same four layers in order:

```
Firestore  в†’  Model  в†’  Service  в†’  Provider  в†’  Screen / Widget
```

### Layer 1 вЂ” Model (`lib/models/`)

Raw Firestore document data is converted into a typed Dart object. For example, `ListingModel.fromFirestore(doc)` maps the document fields into strongly-typed properties. The model also has a `toMap()` method for writing back to Firestore, and a `copyWith()` method for creating updated copies without mutation.

### Layer 2 вЂ” Service (`lib/services/`)

The service classes are the only place in the entire app that talk directly to Firebase.

- `AuthService` вЂ” wraps Firebase Auth for sign-up, sign-in, sign-out, email verification, bookmark toggling, and profile creation
- `FirestoreService` вЂ” wraps all Firestore operations: streaming listings, streaming reviews, creating/updating/deleting listings, and adding reviews
- `LocationService` вЂ” wraps the Geolocator package for requesting GPS permission and returning the device's current position

No screen or widget ever imports a service directly. Services are only accessed through providers.

### Layer 3 вЂ” Provider (`lib/providers/`)

Riverpod providers sit between services and the UI. They subscribe to the service streams and expose the data reactively. When Firestore data changes, the provider automatically notifies every widget watching it.

- `authStateProvider` вЂ” streams the current Firebase Auth user (logged in or null)
- `userProfileProvider` вЂ” streams the current user's Firestore profile document
- `allListingsProvider` вЂ” streams all listings, but only after auth is confirmed
- `userListingsProvider` вЂ” streams only listings created by the current user
- `filteredListingsProvider` вЂ” derives a filtered list from `allListingsProvider` based on the active search query and category filter
- `reviewsProvider(listingId)` вЂ” streams reviews for a specific listing
- `listingNotifierProvider` вЂ” handles create, update, and delete operations with loading and error states
- `authNotifierProvider` вЂ” handles sign-up, sign-in, and sign-out with loading and error states

### Layer 4 вЂ” Screen / Widget (`lib/screens/`, `lib/widgets/`)

Screens use `ref.watch()` to subscribe to providers. They never call Firestore or services directly. When the provider's data changes вЂ” for example when a new listing is added by another user вЂ” the widget rebuilds automatically with the latest data.

For example, the Directory screen watches `filteredListingsProvider`. When the user types in the search bar, `listingFilterProvider` updates, which causes `filteredListingsProvider` to recompute, which causes the list to rebuild вЂ” all without a single database call.

## Navigation Structure

The app has two navigation levels.

### Bottom Navigation (main tabs)

The `HomeShell` widget manages four tabs using an `IndexedStack` so each tab keeps its state when switching:

| Tab | Screen | Purpose |
|-----|--------|---------|
| 1 | `DirectoryScreen` | Browse, search, and filter all listings |
| 2 | `MyListingsScreen` | View and manage your own listings |
| 3 | `MapScreen` | See all listings as map markers |
| 4 | `SettingsScreen` | Profile, name edit, sign out |

### Authentication Gate

`main.dart` contains `_AuthGate`, which watches `authStateProvider`. If the user is null it shows `LoginScreen`. If logged in it shows `HomeShell`. This transition is automatic вЂ” no manual navigation calls needed.
